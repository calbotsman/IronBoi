# Firebase configuration files

This directory holds the per-build-configuration `GoogleService-Info.plist` variants. A build phase script in the IronBoi target picks the right one based on `$CONFIGURATION` and copies it into the `.app` bundle as the canonical `GoogleService-Info.plist` that the Firebase SDK reads at launch.

## How the swap works

```
$CONFIGURATION    Source plist                          Bundle name
─────────────     ────────────────────────────────      ─────────────────────────
Debug             GoogleService-Info-Staging.plist  →   GoogleService-Info.plist
Release           GoogleService-Info-Prod.plist     →   GoogleService-Info.plist
```

The script lives in `project.yml` under `preBuildScripts`. It runs at the start of every build, copies the right source file over the canonical bundle name, and warns if the prod plist still contains placeholder sentinels.

The canonical `IronBoi/GoogleService-Info.plist` is `.gitignore`'d — it's generated by the script every build. **Don't commit it.**

## First-time prod setup

When you create the `ironboi-prod` Firebase project:

1. In Firebase Console → Project settings → General, add an iOS app with bundle id `com.thecombinationrule.ironboi`.
2. Download the `GoogleService-Info.plist` Firebase generates.
3. **Replace this directory's `GoogleService-Info-Prod.plist`** with the downloaded file (overwrite — keep the filename).
4. Commit the replacement. Now Release builds will work end-to-end against prod.

Until step 3 is done, the script will emit a build warning:

```
warning: GoogleService-Info-Prod.plist still contains REPLACE_WITH_PROD_* sentinels.
         Release archives will fail at first Firebase API call. See
         ios/IronBoi/IronBoi/Firebase/README.md.
```

Debug builds work without any prod setup.

## What if I need a third configuration?

Add a new `GoogleService-Info-<Config>.plist` here, then extend the `case "$CONFIGURATION"` block in `project.yml`'s preBuildScripts. The script falls back to staging if the configuration isn't recognized — useful for ad-hoc TestFlight builds that should still talk to staging.

## Why not bundle both files directly?

Apple's reviewer guideline (and the Firebase SDK) expects exactly one `GoogleService-Info.plist` at the root of the `.app` bundle. Shipping both confuses the SDK and would let observers see the staging credentials in a Release build.
